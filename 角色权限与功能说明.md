# 在线拍卖系统 - 角色权限与功能说明（按当前系统与数据库）

本文档依据当前后端 `RoleCheckHelper`、`WorkOrderApiController`、`AuctionWorkOrderServiceImpl`，Vue 路由 `OnlineAuction-vue/src/router/index.js`，以及微信小程序工程 `OnlineAuction-miniprogram` 整理，用于说明角色边界、权限实现与各端入口约束。

---

## 一、角色定义（`auction_user.user_role`）

- **1 - 买方**：前台竞拍与下单用户
- **2 - 卖方**：可发布/管理自己商品的用户
- **3 - 管理员**：后台日常管理与复核
- **4 - 超级管理员**：全局最高权限（与管理员在多数接口上并列）
- **5 - 拍卖师**：拍卖流程与商品审核相关角色
- **6 - 客服**：消息会话与工单客服侧处理
- **7 - 财务**：保证金与订单资金相关处理（后台保证金接口仅财务）
- **8 - 运营**：类目/推荐/内容运营
- **9 - 风控**：异常出价监控、风险用户标记、冻结/解封申请（独立前台路由 `/risk-dashboard`，**不参与** `canAccessAdmin`）
- **10 - 审计**：操作日志、登录日志只读访问（独立前台路由 `/audit-dashboard`）

## 二、当前角色权限概览

> 说明：以下与 `RoleCheckHelper` 及各控制器实际校验一致；若与「直觉上的岗位」不符，以代码为准。

### 1）买方（1）

- 前台浏览商品、参与竞拍、代理出价；查看前台公告与个人订单、保证金、消息中心
- 留言板（发布/查看/编辑/删除自己的留言）
- 工单提交与「我的工单」查看（买方/卖方共用接口）

### 2）卖方（2）

- 拥有买方多数前台能力；新增/编辑/下架自己的商品；本人卖家订单可发货
- 可提交工单与使用消息中心（与买方一致）

### 3）管理员（3）

- 用户管理、卖家资质审核；公告与订单后台管理；商品审核与部分拍卖关键操作
- **工单管理员复核**（接口层与超级管理员并列：`work-order` 的 `/admin-review*`）
- **前台工单处理页 `/work-order`**：路由允许角色 **3 或 6**（与客服共用页面，视图区分）

### 4）超级管理员（4）

- 与管理员并列参与多数「3/4」后台能力；工单复核 API 同样为 **3 或 4**
- **消息中心「全量会话监管」**：`RoleCheckHelper.canViewAllMessageCenter` **恒为 false**，后台不按超管放开全量会话列表（与旧文档表述不同，以代码为准）

### 5）拍卖师（5）

- 商品审核；拍卖延时/流拍/确认成交等（`canAuctioneerManage` / `canConfirmDeal`）
- 可查看后台订单与历史竞拍；**无**用户管理、保证金管理、删除类（`canDelete`）能力

### 6）客服（6）

- 消息中心客服侧；留言后台查看/回复；**工单客服处理**（`/api/work-order/service*`、`/service-process`）
- **前台 `/work-order`**：仅 **3 与 6** 可进入（超级管理员若仅有角色 4 而无 3，则**无法**通过路由进入该页，与复核 API 的 3/4 组合存在差异，部署账号时需注意角色组合）

### 7）财务（7）

- **后台保证金管理**（`canManageDepositAdmin`：**仅角色 7**，不含 3/4）
- 订单资金相关处理（结算、退款等，`canManageOrderAdmin` / `canProcessRefund`：3/4/7）
- 可查看后台订单与历史竞拍

### 8）运营（8）

- 类目、轮播图、推荐位、专场、商品时段等；可参与商品审核与上架状态管理
- 删除类能力通常仅限 3/4（运营无 `canDelete`、`canDeleteNotice` 等）

### 9）风控（9）

- **风控 API**（`/api/risk/*`，`isRiskControl`）
- **前台仅角色 9 可访问** `/risk-dashboard`；后台 SPA 路径 `/admin/*` 前端守卫允许含 9 的角色进入后台壳，但 **`canAccessAdmin` 不包含 9**，依赖该方法的接口仍以服务端校验为准
- 异常出价监控、风险用户池、批量标记风险等级、批量冻结/解封申请（写入工单待复核）

### 10）审计（10）

- **`/api/audit/*`**：`canViewAuditLogs` **仅角色 10**（不是 4/10）
- **前台仅角色 10 可访问** `/audit-dashboard`
- 操作日志、登录日志只读；不具备业务数据写权限

## 三、工单：接口权限与处罚对象规则

### 1）接口角色

| 能力 | 角色（`RoleCheckHelper` / 控制器） |
|------|-----------------------------------|
| 用户提交工单、我的工单 | 1、2 |
| 客服列表与处理 | `isCustomerService` → **6** |
| 管理员复核列表与提交复核 | **3、4**（`WorkOrderApiController`） |

### 2）前台路由 `/work-order`

- 守卫：**仅当 `userRole` 包含 3 或 6** 时允许访问（见 `router/index.js`）。
- 若账号需要「超管打开工单页」，需同时赋予 **3**（或调整产品/路由策略）。

### 3）处罚对象推断（服务端生效，无需手填目标用户）

- **`work_type = message`**：处罚对象为**被投诉留言的发布者**（由 `relatedId` 关联留言表取 `user_id`）。
- **`work_type` 为商品/订单/竞价/其他等投诉类**：处罚对象为**发起工单的用户**（工单上的 `user_id`，即投诉人）。
- **`work_type = risk`**：处罚对象为工单登记的 **`penaltyTargetUserId`**，或与风控登记一致的 **`relatedId` 目标用户**（风控批量申请逻辑）。

接口请求体若携带 `penaltyTargetUserId`，业务上以服务端解析结果为准。

## 四、风控模块现状

### 1）异常出价监控

- 数据来源：`auction_record`
- 关键字段：`risk_rule_type`（1~5）、`abnormal_type`（历史口径）
- 支持商品维度筛选与异常类型筛选（含「全部类型」）；不按买家 ID 筛选监测列表。

### 2）恶意参拍 / 围标刷价汇总

- 不按商品 ID 收窄时可为全局统计（与当前报表实现一致）。
- 恶意参拍：窗口内按 `risk_rule_type in (1,2,3)` 汇总；围标刷价依赖同商品同 IP、去重买家数等规则。

### 3）风险等级与冻结解封

- 从「有风险行为用户」列表多选，避免手输用户 ID。
- 批量标记风险等级：更新 `auction_user.risk_level`。
- 批量冻结/解封：生成 `work_type='risk'` 工单，待管理员复核后生效。

## 五、数据库字段与权限相关说明

### 1）用户与风险

- `auction_user.user_role`：角色编号（可多角色逗号拼接字符串）
- `auction_user.user_status`：账号状态
- `auction_user.risk_level`：风险等级（0 正常递增）

### 2）出价风控

- `auction_record.risk_rule_type`、`abnormal_type`、`bid_ip` 等用于风控筛选与围标判定。

### 3）工单

- `auction_work_order.work_type`：`risk`、`message`、`goods`、`order`、`bid`、`other` 等（具体以前端提交与后端校验为准）。
- 风控提交的冻结/解封：`work_status` 进入待复核，管理员审核通过/驳回。

### 4）操作日志

- 表：`auction_oper_log`
- 审计接口 `/api/audit/*` 对该表只读分页查询。

## 八、微信小程序端（`OnlineAuction-miniprogram`）

### 1）定位与会话

- 小程序面向 **买方** 的浏览、竞拍、查看商品等核心功能。**不包含**后台管理、工单、风控台、审计台、商品发布、订单管理、消息中心、留言板等模块。
- 小程序仅作为竞拍入口，商品发布、订单管理、资金操作等一切售卖行为只能在网页端完成。
- **登录**：新用户通过微信授权后输入手机号和设置密码完成注册。已注册用户通过微信授权后输入手机号和密码登录。小程序通过响应头 **`Set-Cookie: JSESSIONID=...`** 保存会话并复用。
- **请求携带会话**：`miniprogram/utils/request.ts` 将 `JSESSIONID` 存入本地并在后续请求的 **`Cookie`** 头中发送，保证 `/api/mp/*` 与 **`/api/OnlineAuction/auctionRecord/bid`** 等与 Web 共用会话。

### 2）页面结构（`miniprogram/app.json`）

| 类型 | 路径 | 说明 |
|------|------|------|
| Tab | `pages/home/home` | 首页商品列表浏览 |
| Tab | `pages/special/special` | 专场列表 |
| Tab | `pages/my-goods/my-goods` | 我的竞拍（查看竞得的商品） |
| Tab | `pages/profile/profile` | 个人中心（账号信息、退出登录） |
| 普通页 | `pages/login/login` | 微信授权 + 手机号登录 |
| 普通页 | `pages/goods-detail/goods-detail` | 商品详情（图/视频、出价、竞拍记录） |
| 普通页 | `pages/special-goods/special-goods` | 专场下商品列表 |
| 普通页 | `pages/set-password/set-password` | 设置密码（可选，开通网页端登录） |

多数 Tab 页在 **`onShow`** 中检测本地/`globalData` 用户，未登录则 **`wx.reLaunch` 到登录页**。

### 3）小程序专用接口（`/api/mp`，`MpApiController`）

除 **`/auth/wechat-login`** 外，下列接口均要求 **`getCurrentUserId` 非空**（即已登录），**未再按职员角色枚举拦截**——与 Web 端 `RoleCheckHelper` 精细分工不同，小程序接口层只做「登录与否」校验。小程序仅支持商品浏览和竞拍，不包含商品发布、订单管理、资金操作等后台功能。

| 方法与路径 | 作用 |
|------------|------|
| `POST /auth/wechat-login` | 微信登录或注册并建立 Session |
| `GET /home/goods` | 首页商品分页（关键词等） |
| `GET /special/list` | 上架中的专场列表 |
| `GET /special/{id}/goods` | 专场商品列表 |
| `GET /goods/{id}` | 商品详情（含浏览计数、`files` 等媒体） |
| `GET /user/profile` | 当前用户资料 |
| `POST /user/set-password` | 设置密码（可选，开通网页端登录） |

具体实现见：`OnlineAuction-boot/.../controller/mp/MpApiController.java`。前端封装见：`miniprogram/utils/mp-api.ts`。

### 4）竞拍相关（与 Web 共用接口）

| 前端封装 | HTTP | 说明 |
|----------|------|------|
| `submitBid` | `POST /api/OnlineAuction/auctionRecord/bid` | 出价，依赖 Session；服务端 `AuctionRecordController` 当前仅校验登录与业务规则 |
| `getBidRecordsByGoodsId` | `GET /api/OnlineAuction/auctionRecord/list?goodsId=&limit=` | 商品竞拍记录列表；控制器侧**不要求登录** |

### 5）小程序内的「角色」表现（前端逻辑）

- 商品详情页（`goods-detail.ts`）：**出价能力**在用户已登录前提下，还要求 **`userRole` 包含 `1` 或 `2`**（或 `isBuyer` / `isSeller` 为真）；否则提示「当前账号无买方/卖方角色，无法参与竞拍」。另限制：**不能竞拍本人发布的商品**，且须处于竞拍时间窗口且 `goodsStatus` 为竞拍中。
- 个人中心仅展示账号信息和 `userRole` 字符串；若账号仅为内部职员（3~10）而无 1/2，仍可登录小程序并使用需登录的 `/api/mp` 列表，但**竞拍按钮会按上述规则禁用**。

### 6）配置说明

- `miniprogram/app.ts` 中 **`globalData.baseUrl`**（默认示例 `http://127.0.0.1:8080/api`）与 **`uploadBase`**（静态资源域名）须按部署环境修改，并与后端、合法 request 域名配置一致。
